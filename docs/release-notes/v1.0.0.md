# DevMemory v1.0.0 Release Notes

## Release Summary

DevMemory v1.0.0 is the first professional private-deployment release target for local or intranet use. It packages the current course knowledge base, document ingestion, DeepSeek-backed Q&A, generated study materials, mind maps, progress tracking, health checks, backup/restore guidance, and release rehearsal process into a single-machine Docker Compose distribution.

This release is intended for personal use or small trusted teams. It is not a public SaaS release and does not include billing, open registration, multi-tenant administration, or organization role management.

## Deployment Notes

The default private deployment path is Docker Compose:

```powershell
copy .env.production.example .env.production
docker compose --env-file .env.production -f docker-compose.prod.yml up --build -d
```

Only the frontend entrypoint should be exposed to users. Backend API calls should flow through the frontend proxy, and PostgreSQL should remain private to the Compose network.

Before deploying, operators must set strong production values for access token secret, default password, PostgreSQL password, CORS origins, upload directory, upload limits, rate limits, and DeepSeek API key if online AI output is required. Without a DeepSeek API key, DevMemory starts in offline placeholder AI mode.

## Verification Evidence

Before creating the final `v1.0.0` tag, record current evidence for:

- Backend tests: `cd backend; ..\.venv\Scripts\python.exe -m pytest tests`.
- Frontend tests and build: `cd frontend; npm.cmd test; npm.cmd run build`.
- Release tool tests: `python -m pytest scripts`.
- Release rehearsal: `powershell -ExecutionPolicy Bypass -File scripts\release-rehearsal.ps1`.
- Restore evidence: `python scripts\restore_drill_report.py --output release-evidence\restore-drill.md`.
- Tag readiness: `python scripts\release_tag_check.py --version v1.0.0`.
- GitHub Actions: backend, frontend, release tools, and Docker jobs all green on `main`.
- Restore drill: database and upload backups restored in a clean environment, followed by smoke verification.

The generated release rehearsal report belongs under `release-evidence/` and should be archived with the release evidence, not committed.

## Known Risks

- The v1.0 account model is a single configured default user. Team registration, role management, and per-user permissions are deferred.
- Rate limits are in-memory and reset on backend restart, which is acceptable for a single-node private deployment.
- Offline hash embeddings are suitable for local development and fallback testing; production operators should validate retrieval quality with representative documents.
- DeepSeek availability, latency, and output quality depend on the configured external service.
- Clean-environment restore commands remain deliberately manual because database and upload restores are destructive.

## Deferred to v1.1

- Password change and token revocation flow.
- Multi-user or team-aware account model.
- Worker heartbeat history and richer job metrics.
- Model-output evaluation report artifacts.
- Guided first-run checklist and dismissible in-product tips.
- External object storage and multi-node deployment guidance.
